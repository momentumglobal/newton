Platform Overview

Newton is a recruitment reporting and workforce management platform built for Momentum Global. It is a static web application hosted on GitHub Pages, with Microsoft Azure AD for authentication and SharePoint as its data backend.

There is no server-side logic. All data access is performed client-side via the Microsoft Graph API using an authenticated MSAL token. Role-based access control is enforced at the UI layer using role data fetched from SharePoint on sign-in.

Mobile App

Newton is installable on phones as a Progressive Web App (PWA). It is the same web application served from mobile.html, not a separate native build — so it shares one codebase, one data model and one auth flow with the desktop site. Earlier prototyping used a Flutter WebView wrapper; this was retired in favour of the PWA, which runs in the device browser engine and avoids embedded-WebView authentication issues.

Installing

On the phone, open momentumglobal.github.io/newton/mobile.html in Chrome (Android) or Safari (iOS), then choose "Add to Home screen" / "Install app". A Newton icon is added to the home screen and launches standalone (no browser address bar). Sign-in uses the same Microsoft account as the desktop site.

How it works

PieceFilePurpose
Manifestmanifest.webmanifestDeclares the app name, icons (navy Momentum mark, 192/512px), standalone display and #090546 theme. Makes the site installable.
Service workersw.jsNetwork-first and caches no app code, so there is never stale JS after a commit. Ignores cross-origin and non-GET requests, so Microsoft login and SharePoint writes are never intercepted.
Mobile entrymobile.htmlRegisters the service worker, sets a newton_mobile localStorage flag (so desktop pages redirect back to mobile after the login round-trip), and loads the mobile script stack.

Mobile shell & navigation

After login the app lands on a Home launcher — a role-aware grid of modules. A top-bar module switcher dropdown moves between modules; each module has its own bottom-nav tabs. Both are driven by CONFIG.OS_MODULES (single source of truth) filtered through MOBILE_MODULES — a module appears only if it has a built mobile view and the user's role allows it. Command Centre is intentionally excluded from mobile.

Access

Mobile access is restricted to Talent Partners, Delivery Managers and Admins. Per-module visibility follows the same CONFIG.OS_MODULES roles as desktop — e.g. a Talent Partner does not see Sales but does see Market Analytics; Scorecards are role-scoped (TP sees own, DM sees their projects' TPs, Admin sees all).

Adding a module to mobile

Mirror the desktop convention: create a mobile-<module>.js renderer that reuses the existing data/calculation functions, add the module key to MOBILE_MODULES, add a MOBILE_NAV entry and default view, route its views in mobileRenderView(), and load the script in mobile.html. No Flutter rebuild is ever required — the installed PWA picks up changes on next load.

Script Load Order

All HTML files must load scripts in this order. Deviating causes "X is not defined" errors because later scripts depend on globals defined by earlier ones.

Critical: utils.js must always load before api.js. api.js calls functions defined in utils.js (ghost mode helpers, button loading state). This applies to every HTML file including admin.html and index.html.

Field normalisation

SharePoint's Title field and the typo field Yeare are aliased to meaningful names in FIELD_ALIASES in api.js. All reads go through normaliseFields() so the application always sees consistent field names regardless of SharePoint's internal naming.

CCStatus alias: The CCStatus entry in FIELD_ALIASES must be CCStatus: {}. Using { RAG: 'RAG' } causes normaliseFields() to delete the RAG field on every read.

Pagination

getItems() handles SharePoint pagination automatically via @odata.nextLink. All pages of results are fetched and concatenated before returning.

Performance rules

  • Always fetch independent lists in parallel using Promise.all() — never await sequentially.
  • For company-scope views, use getAllRoles() — never loop getRolesForProject() per project.
  • Dashboard period-filter changes use cached window._last* data — no re-fetch.
Write Flow

All writes go through createItem(), updateItem(), or deleteItem() in api.js. Each write automatically invalidates the cache for the affected list.

Cache invalidation: Writes only invalidate the specific list written to. If a write affects data shown via another list, that cache expires naturally within 30 seconds.

Error handling

All write functions are wrapped in try/catch in the form submission layer. On error: button loading state is cleared and an error message is displayed inline. No page navigation occurs on failure.

Coding Conventions

Established during the v1 refactor audit (June 2026). Apply to all future Newton build work.

Single source of truth

Any value referenced in more than one place lives in exactly one file. CONFIG.OS_MODULES is the canonical example — it defines the module switcher for all nav files. CONFIG.UTILISATION_THRESHOLDS defines utilisation RAG thresholds for both the People Dashboard and the Command Centre; CONFIG.REVENUE_THRESHOLDS defines revenue RAG bands for the Sales Revenue Tracking chart, the CC Revenue tile and the homepage Revenue pill. Adding a new module means editing config.js only.

File responsibilities — hard rules

  • Never put UI helpers in api.js. Never put API calls in utils.js.
  • api.js owns all Graph API calls. If it touches the network, it lives here.
  • utils.js owns pure functions with no network I/O — formatting, calculations, button state, ghost helpers.
  • nav-core.js owns the sidebar renderer. Do not write sidebar HTML in module nav files.
  • *-app.js files handle init only — MSAL redirect, role resolution, first navigation.
  • analytics.js owns shared analytics logic — isRoleFlagged(), ACTIVE_STAGES, STAGE_ORDER, computeVelocityScore(). Never duplicate these in page scripts.

CSS over inline styles

  • Conditional styling uses CSS classes, not inline style="" attributes.
  • Scope classes to their context: .dash-panel .row-age-critical td prevents bleed into unrelated tables.
  • Colour values live in style.css only. JavaScript never hardcodes hex colours.

Dynamic over hardcoded

Time-sensitive values are derived at runtime. Example: PLACEMENT_YEARS uses Array.from({length:4}, (_, i) => new Date().getFullYear() - i) — it never needs updating. Utilisation thresholds use CONFIG.UTILISATION_THRESHOLDS — never hardcode 0.85 or 0.75 in page scripts.

No duplicate logic

  • If the same function exists in more than one file, extract it to a shared file.
  • Never re-sort data already sorted by the API layer.
  • Never re-fetch data already cached in a window._ variable.
  • isRoleFlagged() is defined once in analytics.js and used by cc-pages.js, analytics-pages.js, and index.html — do not redefine it locally.

Nav optimisation

Use updateNavActiveLink(page) on navigation — not a full renderNav() rebuild. The full render runs once on init only. This prevents sidebar flicker and unnecessary DOM work on every page click.

Adding a New Module

Follow this checklist in order. Steps 1 and 6 are the most commonly forgotten.

  • config.js — Add entry to CONFIG.OS_MODULES with key, name, icon, href, live, and roles array. This is the only place the module switcher is defined.
  • [module]-router.js — Create page registry. Define PAGES object, canAccess(), and getAccessiblePages().
  • [module]-nav.js — Create nav wrapper. Call renderModuleNav() with subtitle, currentModuleKey, toggleFn, pages, currentPage, role, navigateFn, userGuideHref. Add toggle function and navigateTo[Module]().
  • [module]-app.js — Handle MSAL redirect, call getEffectiveRole(), redirect if role not permitted, call ensureUserRegistered(), render nav, navigate to first page. Set document.getElementById('app-shell').style.display = 'flex' after auth.
  • [module]-pages.js and [module]-forms.js — Create page renderers and form logic as needed.
  • [module].html — Follow script load order exactly: config → auth → utils → api → [module]-router → nav-core → [module]-nav → [module]-forms → [module]-pages → [module]-app. Load analytics.js before app.js if the module uses isRoleFlagged() or other shared analytics functions.
  • admin.html sidebar — Add <a class="admin-nav-link"> entry if the module needs admin panel access.
  • os-admin.js — Update role/access lists if the new module has different access requirements.
The Report Builder's REPORT_PANELS registry in dashboard.js is the pattern for adding new report panel types — each panel is a function that takes data and a period and returns HTML.
Ghost Mode

Ghost Mode allows admins to temporarily view Newton as a different role, for testing or training video creation. Available from Newton OS Admin → Ghost Mode tab.

→

→
Banner rendered by nav-core.js

Storage keys (sessionStorage)

KeyValueSet by
newton_ghost_roleRole string e.g. "delivery_manager"setGhostRole() in utils.js
newton_ghost_projectSharePoint project ID (string)setGhostProject() in utils.js

Exiting ghost mode

Exit from the banner button or the Ghost Mode tab. Both call clearGhostRole() which removes both sessionStorage keys. Sign-out also clears ghost mode via sessionStorage.clear().

Ghost mode overrides role resolution but does not create a real session as that role. Actions taken while ghosting (form saves etc.) are attributed to your real admin account in SharePoint.

Backlog & Future Work

Known improvement opportunities and architectural considerations for future development.

Architectural considerations (future builds)

AreaConsideration
Data volumegetItems() fetches all records and filters client-side. As data grows (especially WeeklyActivity), consider adding server-side OData filters to reduce payload size.
Multi-project usersDM and TP users can be assigned to multiple projects. The scoping model handles this but the UX could be more prominent for users with many assignments.
Real-time updatesThe 30s API cache means two users editing simultaneously may see stale data briefly. For higher-traffic scenarios, consider a shorter TTL or manual refresh triggers.
Report Builder exportPDF export uses window.print(). Server-side generation would give more reliable formatting across browsers.
Offline / error statesGraph API failures show a basic error message. A structured error boundary with retry logic would improve resilience on poor connections.
Module expansionThe module pattern (router / nav / app / pages / forms) is repeatable. New modules can be added without touching shared platform code — only config.js needs updating for the switcher.
Command Centre alertsFuture: push notifications or email digest when a CC tile flips to Red.
CCStatus freshnessCCStatus is only updated when an Admin or Leadership user loads the CC or triggers a homepage refresh. Consider a scheduled write from the CC page to keep state fresh.
